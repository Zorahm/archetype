

# Archetype — Полная концепция мода

## Система классов для Minecraft

### Forge + Fabric | Minecraft 1.20.1+

---

# Часть 1. Идея

Мод добавляет систему классов. Каждый класс меняет характеристики игрока, даёт уникальные активные и пассивные способности, накладывает ограничения. Классы определяются через JSON-файлы — без написания Java-кода. Сервер может добавлять свои классы через датапаки. Другие моды могут добавлять новые типы способностей программно.

Главный принцип: **баланс через компромисс**. Каждый бонус компенсирован штрафом. Вампир силён ночью, но горит днём. Голем неубиваем, но тонет. Фантом проходит сквозь стены, но хрупок.

# Часть 2. GUI

## Философия

Два экрана. Витрина и досье. Минимум кликов, максимум информации.

### Элементы карточки

Каждая карточка содержит:

- Цветная рамка — цвет класса из JSON
- Иконка — крупная, пиксельная, 32×32 или 48×48
- Название — одно-два слова под иконкой
- Маленький значок роли в углу — мечи (урон), щит (танк), перо (мобильность), глаз (утилити)

### Поведение при наведении

Курсор наводится на карточку:

- Карточка плавно увеличивается (масштаб ×1.1, время анимации 200мс)
- В нижней части экрана появляется Lore-блок:
    - Атмосферная цитата (1–2 строки курсивом)
    - Ключевая механика одной строкой: «Горит на солнце. Пьёт кровь. Летает ночью.»
    - Три маркера силы: иконки ⚔️🛡️💨 с заполненными/пустыми делениями — быстрая оценка роли без чтения текста

### Клик по карточке

Левая кнопка мыши → переход к экрану-досье выбранного класса.

### Масштабирование под количество классов

| Классов | Сетка | Скролл |
|---------|-------|--------|
| 3–6 | 3×2, карточки крупные | Нет |
| 7–15 | 5×3, карточки средние | Нет |
| 16–24 | 6×4, карточки мелкие | Нет |
| 25+ | 6×4 + вертикальный скролл сетки | Да |

При количестве классов более 20 — над сеткой появляются фильтры по роли. Не отдельные вкладки, а кнопки-переключатели, которые скрывают/показывают карточки прямо в сетке.

```
[Все] [⚔️ Урон] [🛡️ Танк] [💨 Мобильность] [🔮 Утилити]
```

### Режим перерождения

Когда витрина открывается через предмет смены класса или команду у игрока с уже выбранным классом:

- Заголовок меняется на «ПЕРЕРОЖДЕНИЕ»
- Текущий класс обведён золотой рамкой с пометкой «Текущий»
- Остальное — без изменений

### Кнопка «Случайный»

Внизу экрана. Выбирает случайный класс, сразу открывает его досье.

### Клавиатурная навигация

- Стрелки — перемещение по сетке (активная карточка подсвечена)
- Enter — эквивалент ЛКМ
- ESC — закрыть, если класс уже выбран. Если класс не выбран — нельзя закрыть
- Tab — переключение фильтров (если есть)

### Секции сверху вниз

**Шапка.** Иконка класса (крупнее чем на витрине), название, атмосферная цитата курсивом. Название окрашено в цвет класса.

**Характеристики.** Визуальные полоски. Средний уровень (ванильный игрок) — середина полоски. Выше нормы — полоска длиннее, зелёный цвет заполнения. Ниже нормы — полоска короче, красный цвет. На уровне нормы — серый. Справа — числовое значение и разница с ванилой в скобках.

Полоски показываются для всех атрибутов, у которых модификатор ≠ 0. Если класс не трогает броню — строка брони не показывается вообще.

**Активные способности.** Каждая способность — блок:
- Слева: клавиша в квадратных скобках (берётся из текущих настроек бинда)
- По центру: название
- Справа: кулдаун в секундах
- Под названием: описание 1–2 строки серым текстом
- Если разблокируется на определённом уровне — плашка «Уровень N» рядом с кулдауном

При наведении курсора на блок способности — расширенный тултип с точными числами: урон, радиус, длительность, стоимость ресурса. Для тех, кому нужна точная информация.

**Пассивные особенности.** Список с маркерами:
- ✔ Зелёный — бонус
- ✘ Красный — штраф

Порядок: сначала бонусы, потом штрафы. Каждый пункт — одна строка: название + тире + краткое пояснение.

**Ресурс.** Если у класса есть уникальный ресурс (кровь, мана, ярость) — полоска с максимальным значением, описание в 1–2 строки: как тратится, как восполняется.

Если ресурса нет — секция не отображается.

**Кнопка «Выбрать класс».** Внизу, крупная, по центру. После нажатия:
- Если это первый выбор (класса нет) — класс назначается сразу
- Если это перерождение (класс есть) — появляется подтверждение: «Ты потеряешь все способности [текущего класса]. Продолжить?» с кнопками Да/Нет прямо на этом же экране, без нового окна

**Кнопка «Назад».** Левый верхний угол. Возвращает на витрину.

### Скролл

Содержимое досье длиннее экрана. Скроллится колёсиком мыши. Скролл плавный с инерцией — не попиксельный. Ограничения: не уходит выше первой секции и ниже последней.

### Анимация перехода витрина → досье

Карточка, по которой кликнули, «раскрывается» на весь экран:
1. Карточка начинает расти, остальные карточки уходят в прозрачность
2. Через 300мс — содержимое досье появляется

Обратно (кнопка «Назад»): досье «сжимается» обратно в карточку, остальные карточки возвращаются.

Если анимации отключены в конфиге — мгновенный переход.

### Звуки

- Наведение на карточку — тихий клик
- Клик по карточке — звук «развернулся свиток»
- Подтверждение выбора — торжественный аккорд
- Перерождение — тревожный звук, отличный от первого выбора

## HUD в игровом процессе

После выбора класса на экране появляется компактная панель:

```
[R]🩸  [V]🦇  [G]🔒       🩸 ████████░░ 73/100
```

- Слоты способностей: иконка + клавиша сверху
- При кулдауне: иконка затемняется сверху вниз, в центре — оставшиеся секунды
- Заблокированная уровнем способность — замок
- Полоска ресурса справа (если есть)
- Цвет полоски = цвет ресурса из JSON

Позиция HUD настраивается в конфиге: справа от хотбара, слева от хотбара, сверху по центру.

## Экран информации (в процессе игры)

Клавиша J (настраиваемая) → открывает досье текущего класса. Та же страница, что при выборе, но вместо кнопки «Выбрать»:
- Текущий уровень класса
- Полоска опыта до следующего уровня
- Текущее значение ресурса

# Часть 3. Определение классов

## Формат JSON

Каждый класс — один JSON-файл в `data/<namespace>/archetype_classes/`.

### Структура файла

```
Обязательные поля:
  id              — уникальный идентификатор (namespace:name)
  name            — ключ перевода для названия
  description     — ключ перевода для описания
  icon            — путь к текстуре иконки
  color           — HEX-цвет класса

Необязательные поля:
  category        — роль (damage/tank/mobility/utility)
  lore            — массив ключей перевода для атмосферных строк
  attributes      — модификаторы базовых атрибутов
  conditional_attributes — модификаторы по условию
  passive_abilities      — пассивные способности
  active_abilities       — активные способности
  resource               — уникальный ресурс
  incompatible_with      — список несовместимых классов
  size_modifier          — изменение размера модели
  death_drops            — предметы при смерти
```

### Атрибуты

Модификаторы стандартных атрибутов Minecraft. Указываются: ID атрибута, операция (addition/multiply_base/multiply_total), значение. Работают через ванильную систему AttributeModifier. Совместимы с `/attribute`.

Поддерживаются и кастомные атрибуты мода: `archetype:blood_capacity`, `archetype:mana_capacity`, `archetype:cooldown_reduction`, `archetype:ability_range`, `archetype:resource_regen_rate`.

### Условные атрибуты

Модификаторы, применяемые при выполнении условия. Условие — отдельный объект с типом и параметрами. Встроенные типы условий:

| Тип условия | Параметры | Пример |
|---|---|---|
| time_of_day | range [from, to] | Ночь: [13000, 23000] |
| health_below_percent | percent | Менее 30% HP |
| health_above_percent | percent | Более 80% HP |
| in_water | — | Игрок в воде |
| underwater | — | Голова под водой |
| under_open_sky | — | Нет блока над головой |
| in_dimension | dimension | minecraft:the_nether |
| in_biome_tag | tag | minecraft:is_ocean |
| is_sneaking | — | Зажат Shift |
| is_sprinting | — | Бежит |
| on_fire | — | Горит |
| has_item | item, slot | Предмет в руке/слоте |

Условия комбинируются: `and`, `or`, `not`.

### Пассивные способности

Типы пассивок — готовый набор. Автор контента выбирает тип и настраивает параметры.

| Тип | Что делает | Ключевые параметры |
|---|---|---|
| sun_damage | Урон на солнце | damage_per_second, ignore_if_helmet, ignore_if_underground |
| night_vision | Ночное зрение | condition (когда включается), amplifier |
| food_restriction | Ограничение еды | allowed_foods, tag_allowed, on_eat_forbidden (эффект) |
| natural_regeneration_disabled | Нет естественной регенерации | — |
| undead_type | Тип «нежить» | smite_vulnerability, healing_potions_damage, damage_potions_heal |
| water_vulnerability | Урон от воды | damage_per_second, rain_damage, rain_damage_per_second |
| sink_in_water | Тонет | cannot_swim, walk_on_bottom, walk_speed_underwater |
| no_fall_damage | Нет урона от падения | — |
| fire_immunity | Иммунитет к огню | — |
| custom_diet | Кастомная еда | food_items (что можно есть), food_value, saturation |
| mob_neutral | Нейтральность мобов | mob_types, always_friendly |
| magnetic_pull | Притягивание предметов | item_pull_radius, item_pull_speed |
| thorns_passive | Отражение урона | reflect_percent, reflect_type |
| lifesteal | Вампиризм | percent_of_damage, only_melee |
| wall_climb | Лазание по стенам | climb_speed |
| slow_fall | Замедленное падение | fall_speed |
| jump_boost | Усиленный прыжок | multiplier |
| breath_underwater | Дыхание под водой | — |
| effect_immunity | Иммунитет к эффекту | effects (список) |

### Активные способности

Типы активных способностей — шаблоны, настраиваемые через JSON.

| Тип шаблона | Механика | Примеры использования |
|---|---|---|
| timed_buff | Нажал → эффекты на N секунд | Stone Skin голема, ускорение берсерка |
| area_attack | Нажал → урон всем вокруг | Ground Slam голема, взрыв мага |
| targeted_effect | Нажал → эффект на сущность, на которую смотришь | Гипноз вампира, проклятие жреца |
| projectile | Нажал → летит снаряд | Огненный шар мага, ледяная стрела |
| dash | Нажал → рывок вперёд | Рывок берсерка, теневой шаг фантома |
| teleport | Нажал → телепортация | Телепорт к точке прицела, обмен с целью |
| morph | Нажал → превращение | Форма летучей мыши, форма волка |
| toggle | Нажал → включилось, нажал → выключилось | Фортификация голема, аура друида |
| summon | Нажал → призвание существа | Призыв скелета, призыв волка |
| self_heal | Нажал → лечение | Регенерация друида, жертвоприношение крови |
| charged | Зажал → копится заряд → отпустил → эффект от заряда | Мощный удар, направленный луч |
| blood_drain | Нажал → высасывает HP из цели в течение N тиков | Укус вампира |

Параметры каждого шаблона: кулдаун, стоимость ресурса, урон, радиус, длительность, эффекты на себя, эффекты на цель, партиклы, звук, условия активации, уровень разблокировки.

Привязка к клавише указывается строкой: `ability_1`, `ability_2`, `ability_3`.

### Ресурс

Уникальная полоска помимо здоровья и голода. Определяется: тип (строка — для перевода), максимум, стартовое значение, естественная убыль в секунду, естественное восполнение в секунду, цвет полоски, иконка, позиция на HUD.

Ресурс тратится способностями. Восполняется действиями (укус вампира возвращает кровь) или пассивно со временем (мана мага).

Если ресурса нет — поле `resource` отсутствует или равно null.

# Часть 4. Мультилоадерная архитектура

## Подход

**Architectury** — прослойка, абстрагирующая различия между Forge и Fabric. Один исходный код, две сборки.

### Структура проекта Gradle

```
archetype/
├── common/          ← Общий код (95% логики)
│   └── src/main/java/com/mod/archetype/
│       └── ...всё, что не зависит от лоадера
│
├── fabric/          ← Fabric-специфичное (5%)
│   └── src/main/java/com/mod/archetype/fabric/
│       ├── ArchetypeFabric.java          ← точка входа
│       ├── FabricClassDataAttachment.java
│       ├── FabricNetworkHandler.java
│       └── FabricEventTranslator.java
│
├── forge/           ← Forge-специфичное (5%)
│   └── src/main/java/com/mod/archetype/forge/
```

### Что уходит в common (общее)

Почти всё:

- Определение класса PlayerClass и его загрузка из JSON
- Реестр классов ClassRegistry
- Определения всех типов способностей
- Фабрика способностей AbilityRegistry
- Система условий (Conditions)
- Менеджер классов ClassManager (логика назначения/снятия)
- Менеджер кулдаунов CooldownManager
- Данные игрока PlayerClassData (структура и сериализация в NBT)
- Команды (регистрация через Brigadier — одинаков на обоих лоадерах)
- GUI: экран витрины, экран досье, HUD-оверлей
- Описание пакетов (что в них содержится)
- Предметы и их логика
- Звуки (регистрация SoundEvent)

### Что уходит в forge

- Точка входа: `@Mod` аннотация, подписка на Forge-шину событий
- Capability Provider: привязка PlayerClassData к игроку через `AttachCapabilitiesEvent`
- Сеть: регистрация пакетов через `SimpleChannel`
- Трансляция событий: `LivingHurtEvent` → вызов общего обработчика, `PlayerTickEvent` → вызов общего обработчика, `PlayerEvent.PlayerLoggedInEvent` → вызов общего обработчика
- Регистрация атрибутов через `DeferredRegister<Attribute>` + `EntityAttributeModificationEvent`
- Регистрация клавиш через `RegisterKeyMappingsEvent`
- HUD рендер через `RenderGuiEvent.Post`

### Что уходит в fabric

- Точка входа: `ModInitializer`, `ClientModInitializer`
- Data Attachment: привязка PlayerClassData через Cardinal Components API или Fabric API Attachment
- Сеть: регистрация пакетов через `ServerPlayNetworking` / `ClientPlayNetworking`
- Трансляция событий: `ServerPlayerEvents.AFTER_RESPAWN`, `ServerTickEvents.END_SERVER_TICK`, `AttackEntityCallback`, `PlayerBlockBreakEvents` — вызов тех же общих обработчиков
- Регистрация атрибутов через `FabricDefaultAttributeRegistry` + `EntityAttributeModifiedCallback`
- Регистрация клавиш через `KeyBindingHelper`
- HUD рендер через `HudRenderCallback`

### Абстракция платформы

В common определяются интерфейсы для всего платформозависимого:

**Интерфейс хранения данных игрока:**
- `getPlayerClassData(Player) → PlayerClassData`
- `setPlayerClassData(Player, PlayerClassData)`

Forge-реализация: через Capability. Fabric-реализация: через Cardinal Components или Attachment API.

**Интерфейс сети:**
- `sendToPlayer(ServerPlayer, Packet)`
- `sendToServer(Packet)`
- `sendToTracking(Entity, Packet)`
- `registerServerReceiver(PacketType, Handler)`
- `registerClientReceiver(PacketType, Handler)`

Forge-реализация: SimpleChannel. Fabric-реализация: Fabric Networking API.

**Интерфейс событий:**
- Architectury Events API покрывает большинство: тик игрока, урон, смерть, вход в мир, респаун
- Если чего-то нет — Mixin в fabric-модуле и подписка на Event в forge-модуле, оба вызывают общий обработчик

**Интерфейс регистрации:**
- Architectury `DeferredRegister` — работает на обоих лоадерах

**Forge:**

|---|---|---|

### GUI — нюансы кроссплатформенности

GUI рисуется целиком в common. Всё что нужно — `Screen`, `GuiGraphics`, `Widget`, `Font` — присутствует в ванильном Minecraft и одинаково на обоих лоадерах.

# Часть 5. Модульная архитектура (пакеты в common)

## Модуль Registry

Загружает JSON-файлы классов при старте мира. Перезагружает по команде `/archetype reload`.

Процесс загрузки:

1. Сканирует `data/*/archetype_classes/` во всех датапаках (включая встроенные в мод)
2. Парсит каждый JSON через Gson
3. Валидирует: существуют ли указанные атрибуты, зарегистрированы ли типы способностей, корректны ли значения
4. Если файл невалиден — логирует ошибку с указанием файла и поля, пропускает, не крашит
5. Складывает валидные описания классов в Map<ResourceLocation, PlayerClass>

При перезагрузке: очищает реестр → загружает заново → для каждого онлайн-игрока: снимает старые модификаторы → применяет из обновлённого описания.

Другой мод может программно зарегистрировать:
- Новый тип пассивной способности
- Новый тип активной способности
- Новый тип условия

После этого JSON-классы (включая добавленные через датапаки) могут использовать эти типы.

## Модуль Core

Центральная логика. Не зависит от лоадера. Получает данные игрока через абстракцию.

### Жизненный цикл класса

**Назначение:**
1. Если есть старый класс — снять (удалить модификаторы, остановить пассивки, очистить кулдауны)
2. Применить модификаторы атрибутов
3. Создать инстансы пассивных способностей, вызвать `onApply` каждой
4. Создать инстансы активных способностей
5. Инициализировать ресурс
6. Применить модификатор размера (если есть)
7. Синхронизировать с клиентом
8. Уведомить другие моды через событие

**Серверный тик (каждый тик для каждого онлайн-игрока с классом):**

| Действие | Частота | Почему |
|---|---|---|
| Уменьшить все кулдауны на 1 | Каждый тик | Точность |
| Тикнуть активные длительные способности | Каждый тик | Плавность |
| Проверить условные атрибуты | Раз в 20 тиков | Условия не меняются часто |
| Тикнуть пассивные способности | Раз в 20 тиков | 1 раз/сек достаточно |
| Обновить ресурс (убыль/восполнение) | Раз в 10 тиков | Достаточная точность |
| Синхронизировать с клиентом | Раз в 20 тиков | Чаще — нагрузка на сеть |

**Смерть и респаун:**
- Данные класса переживают смерть
- После респауна: заново применяются атрибуты (Minecraft сбрасывает их при смерти), заново запускаются пассивки
- Ресурс сбрасывается до стартового значения
- Кулдауны сохраняются

**Снятие класса:**
1. Удалить все модификаторы атрибутов
2. Вызвать `onRemove` каждой пассивки
3. Принудительно деактивировать все активные способности
4. Очистить кулдауны
5. Убрать ресурс
6. Вернуть размер модели
7. Синхронизировать с клиентом
8. Уведомить через событие

## Модуль Abilities

Фабрика способностей. Хранит Map<ResourceLocation, Factory> для пассивных и активных.

### Пассивные

Интерфейс: `onApply(Player)`, `tick(Player)`, `onRemove(Player)`, плюс хуки для событий (`onPlayerHurt`, `onPlayerAttack`, `onPlayerEat`).

Каждая пассивка читает свои параметры из JSON при создании. Хранит их как поля. Логика тика зависит от типа.

Пример логики `sun_damage`: каждый тик → проверить: игрок под открытым небом? → день? → нет шлема (или ignore_if_helmet = false)? → нанести урон → поджечь на N тиков.

### Активные

Интерфейс: `canActivate(Player) → bool`, `activate(Player) → Result`, `tickActive(Player)`, `forceDeactivate(Player)`, `isActive() → bool`.

Подтипы:

- **Мгновенная:** `activate` делает всё и возвращает результат. `tickActive` не вызывается.
- **Длительная:** `activate` запускает процесс, `tickActive` вызывается каждый тик пока `isActive() == true`, через N тиков завершается сама.
- **Toggle:** `activate` переключает состояние. Если уже активна — деактивируется. Если не активна — активируется. `tickActive` тикает пока включена.
- **Заряжаемая:** `activate` начинает зарядку. Каждый тик заряд растёт. Когда клиент отправляет «отпустил кнопку» — вызывается `release(Player, chargeLevel)`.

### Система условий

Переиспользуемые проверки. Один и тот же тип условия используется и в условных атрибутах, и в пассивках (условие для ночного зрения), и в активных способностях (проверка перед активацией).

Интерфейс: `test(Player) → bool`.

Комбинаторы: `and(Condition...)`, `or(Condition...)`, `not(Condition)`.

Парсятся из JSON рекурсивно.

## Модуль Network

Описание пакетов и их обработчиков.

**От клиента к серверу:**

| Пакет | Содержимое | Валидация на сервере |
|---|---|---|
| ClassSelectPacket | ResourceLocation classId | Класс существует, нет кулдауна смены, предмет в руке (если через предмет) |
| AbilityUsePacket | String slotName (max 32 chars) | Класс назначен, способность существует в этом слоте, не на кулдауне, ресурса хватает, условия выполнены |
| AbilityReleasePacket | String slotName | Способность заряжаемая и сейчас заряжается |

**От сервера к клиенту:**

| Пакет | Содержимое | Когда |
|---|---|---|
| OpenClassSelectionPacket | пусто (или byte режим: 0=первый выбор, 1=перерождение) | Первый вход, команда, предмет |
| SyncClassDataPacket | classId, level, resource (current/max), массив кулдаунов (abilityId + remaining + max) | При назначении, каждую секунду, при изменении |
| ClassAssignResultPacket | bool success, optional string failReasonKey | После попытки выбора |
| PlayerClassSyncPacket | UUID playerId, ResourceLocation classId | Другим игрокам, для отображения |

Размер SyncClassDataPacket: ResourceLocation (строка ~30 байт) + int level (4) + float×2 resource (8) + N×(ResourceLocation+int+int) кулдауны (~40 байт на способность). Итого ~100–150 байт. Отправляется раз в секунду. Приемлемо для 100+ игроков.

## Модуль Data

Класс PlayerClassData. Хранит:

- `ResourceLocation currentClassId` — ID текущего класса (nullable)
- `long classAssignedTime` — время назначения
- `int classLevel` — уровень класса (1–20)
- `int classExperience` — опыт
- `float resourceCurrent` — текущее значение ресурса
- `Map<ResourceLocation, Integer> cooldowns` — оставшиеся кулдауны
- `Set<Integer> activeConditionalSets` — какие условные наборы атрибутов сейчас активны
- `Map<ResourceLocation, Boolean> toggleStates` — состояние toggle-способностей

Сериализация в NBT (CompoundTag). Десериализация из NBT. Методы `save()` и `load(CompoundTag)`.

Отдельно: методы `writeSyncData(FriendlyByteBuf)` и `readSyncData(FriendlyByteBuf)` — для сетевой синхронизации. Отправляется только динамическая информация (не описание класса).

Привязка к игроку:
- Forge: через Capability, `AttachCapabilitiesEvent<Entity>`
- Fabric: через Cardinal Components или Data Attachment API

Обе реализации вызывают одни и те же `save`/`load` общего класса.

## Модуль Commands

Регистрация через Brigadier (одинаков на обоих лоадерах, Architectury предоставляет общую точку регистрации).

## Модуль GUI

Целиком в common. Использует ванильные средства: `Screen`, `GuiGraphics`, `Widget`.

Три экрана:
1. `ClassSelectionScreen` — витрина
2. `ClassDetailScreen` — досье
3. `ClassInfoScreen` — информация о текущем классе (в игре, по клавише J)

Один HUD-оверлей: `AbilityHudOverlay` — рисует слоты способностей и полоску ресурса.

Анимации реализуются через `partialTick` и интерполяцию. Каждый анимируемый элемент хранит `currentValue` и `targetValue`. Каждый кадр: `currentValue += (targetValue - currentValue) * speed * deltaTime`.

## Модуль Keybinds

Регистрация клавиш — платформозависимая (в forge/fabric модулях). Обработка нажатий — в common, через абстракцию.

Четыре клавиши:

| Действие | Клавиша по умолчанию | Что делает |
|---|---|---|
| Способность 1 | R | Отправляет серверу AbilityUsePacket("ability_1") |
| Способность 2 | V | Отправляет серверу AbilityUsePacket("ability_2") |
| Способность 3 | G | Отправляет серверу AbilityUsePacket("ability_3") |
| Информация о классе | J | Открывает ClassInfoScreen на клиенте |

Все переназначаемые через настройки управления.

Обработка: каждый клиентский тик проверяет `consumeClick()`. Перед отправкой пакета проверяет: есть ли класс, не на кулдауне ли (клиентский трекер), не открыт ли другой экран. Для заряжаемых способностей: проверяет `isDown()` каждый тик.

# Часть 6. Совместимость

## С ванильными командами

**`/attribute`** — полная совместимость. Модификаторы класса видны, работают по ванильным правилам. Сторонний модификатор стакается с классовым.

**`/effect`** — полная совместимость. Пассивки, дающие эффекты, используют ванильные MobEffect. `/effect clear` снимает, но через секунду класс восстановит.

**`/data`** — данные класса хранятся в Capability, не в ванильном NBT. Доступ — только через команды мода.

**`/kill`, `/gamemode`** — корректно обрабатываются. Смерть не сбрасывает класс. Креатив не ломает способности.

## С другими модами

**Через события.** Мод публикует:
- ClassAssignedEvent — класс назначен
- ClassRemovedEvent — класс снят
- AbilityUsedEvent — способность использована
- AbilityPreUseEvent (cancelable) — перед использованием, можно отменить

**Через API.** Другой мод может:
- Получить класс игрока: `ArchetypeAPI.getPlayerClass(player)`
- Проверить, есть ли у игрока конкретный класс: `ArchetypeAPI.hasClass(player, id)`
- Зарегистрировать тип способности: `ArchetypeAPI.registerAbilityType(id, factory)`
- Зарегистрировать тип условия: `ArchetypeAPI.registerConditionType(id, factory)`
- Программно назначить класс: `ArchetypeAPI.assignClass(player, id)`

API-модуль выделяется в отдельный артефакт (архив), чтобы другие моды могли зависеть от него без зависимости от реализации.

# Часть 7. Хранение и производительность

## Хранение данных

PlayerClassData сохраняется вместе с данными игрока (Capability на Forge, Data Attachment на Fabric). Автоматическое сохранение при выходе, автосохранении мира, `/save-all`.

Размер данных на игрока: ~200 байт NBT. Пренебрежимо мало.

## Серверная нагрузка

Главный потребитель: тик каждого игрока с классом.

**Оценка на 100 игроков:**

| Операция | За тик | Стоимость |
|---|---|---|
| Уменьшить кулдауны (3 числа × 100 игроков) | 300 операций | Бесплатно |
| Тикнуть активные способности (в среднем 5 из 100 используют в моменте) | 5 вызовов | Минимально |
| Проверить условия (раз в 20 тиков, 2 условия × 100) | 10 проверок/тик (в среднем) | Минимально |
| Тикнуть пассивки (раз в 20 тиков, 3 пассивки × 100) | 15 вызовов/тик (в среднем) | Минимально |
| Обновить ресурс (раз в 10 тиков, 100) | 10 вызовов/тик | Минимально |
| Синхронизация (раз в 20 тиков, 150 байт × 100) | ~750 байт/тик (15 кб/сек) | Приемлемо |

Итого: нагрузка незаметна даже на 200+ игроках.

**Что может быть тяжёлым:** пассивки, которые проверяют блоки мира (urон от солнца → проверка skyLight, нахождение в воде → проверка блока). Но это одиночные проверки `level.getBlockState()` — стоимость ничтожна.

## Клиентская нагрузка

HUD-оверлей: 6–8 вызовов `blit` + 4–5 `drawString` + 2–3 `fill`. Стоимость: ~0.01мс на кадр. Нулевое влияние на FPS.

Анимации GUI: интерполяция 10–20 float-значений за кадр. Нулевое влияние.

# Часть 9. Предмет смены класса

**Свиток Перерождения.** Редкость: Epic (фиолетовое название). Стакается до 1. Зачарованный блеск.

Крафт: дорогой. Центральный ингредиент — Звезда Незера или аналогичный по редкости.

Механика:
1. ПКМ → пакет серверу «хочу открыть выбор через предмет»
2. Сервер проверяет предмет в руке → отправляет клиенту «открой витрину в режиме перерождения»
3. Игрок выбирает → пакет серверу
4. Сервер перепроверяет предмет в руке → если есть: тратит предмет, назначает класс, синхронизирует
5. Если предмета уже нет → отклоняет

# Часть 10. Расширяемость

## Как автор контента добавляет класс

Без Java. Только файлы.

1. Создать `data/mypack/archetype_classes/my_class.json` — описание класса
2. Положить иконку в `assets/mypack/textures/gui/class_icons/my_class.png`
3. Добавить переводы в `assets/mypack/lang/en_us.json` и `ru_ru.json`
4. Положить файлы в датапак или ресурспак
5. Перезагрузить: `/archetype reload`

Автор комбинирует готовые типы способностей и условий через JSON. Не пишет код.

## Как мододел добавляет новый тип способности

Нужна Java.

1. Написать класс, реализующий интерфейс PassiveAbility или ActiveAbility
2. В инициализации мода: вызвать `ArchetypeAPI.registerAbilityType("mymod:gravity_control", MyAbility::new)`
3. Теперь JSON-классы могут использовать `"type": "mymod:gravity_control"` с любыми параметрами, которые MyAbility умеет читать из JsonObject

## Как мододел реагирует на события Archetype

Подписка на события через Architectury Events (кроссплатформенная) или через нативные шины (Forge EventBus / Fabric Callbacks).

Доступные события:
- Класс назначен
- Класс снят
- Способность использована
- Перед использованием способности (можно отменить)
- Уровень класса повышен
- Ресурс изменился

# Часть 11. Адвансменты

Встроенные достижения, поддерживаемые кастомным триггером:

```
"Начало пути"           — выбрать первый класс
"Перерождение"          — сменить класс
"Мастер"                — достичь максимального уровня класса
"Испытать всё"          — попробовать каждый класс хотя бы раз
"Первая кровь"          — впервые использовать активную способность
"Идеальный шторм"       — использовать все три способности за 10 секунд
"На пределе"            — использовать способность при ресурсе < 5%
"Против природы"        — вампир пережил рассвет без укрытия
```

Триггер `archetype:class_action` с условиями: тип действия, ID класса, ID способности, уровень. Позволяет датапакам определять свои адвансменты для кастомных классов.
